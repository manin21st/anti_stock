# 소스 코드 정밀 분석 워크플로우

이 워크플로우는 지정된 파일이나 모듈을 정밀 분석하여 **불필요한 코드**, **표준에서 벗어난 패턴**, **성능 저하 요인**을 식별하고 개선안을 제시합니다.

## 1. 분석 대상 확인
사용자가 분석을 요청한 파일을 `view_file` 도구로 읽습니다. (파일이 크면 분할해서 읽습니다.)

## 2. 분석 기준 (Checklist)
코드를 읽으면서 다음 세 가지 관점에서 비판적으로 분석합니다.

### A. 🗑️ 불필요한 로직 (Unused / Dead Code)
- 호출되지 않는 함수나 클래스
- `pass`만 있고 구현되지 않은 스텁(Stub) 코드
- 주석 처리된 채 방치된 레거시 코드 (`legacy`, `deprecated` 키워드 확인)
- 디버깅용으로 남겨진 임시 코드 (`print`, `inject_dummy_data` 등)

### B. ⚠️ 비표준/안전하지 않은 패턴 (Non-Standard)
- **FastAPI**: Pydantic 모델 대신 `request.json()` 직접 파싱, 전역 변수 의존 (`global`), 직접적인 `try-catch` 남발.
- **Python**: 불필요한 중첩, 매직 넘버 사용, 모듈 간 강한 결합.
- **보안**: 하드코딩된 비밀번호나 키.

### C. 🚀 성능 및 구조적 비효율 (Inefficiency)
- 루프 안에서의 DB/API 호출 (N+1 문제)
- 이미 가진 데이터를 다시 조회하는 중복 로직
- 너무 긴 함수 (단일 책임 원칙 위반)

## 3. 보고서 작성 양식 (Korean)
분석 결과는 반드시 **한글**로 작성하며, 아래 형식을 따릅니다.

---
### 📊 [파일/모듈명] 분석 보고서

**1. ⚠️ 표준 방식과 다른 패턴 (Non-Standard)**
- **[발견 항목]**: (구체적인 위치와 이유)
- **[표준 가이드]**: (권장되는 모범 사례)

**2. 🗑️ 사용 안 하거나 불필요한 로직 (Unused/Bloated)**
- **[함수/변수명]**: (삭제 권장 사유)

**3. 🚀 성능 및 구조 개선 포인트**
- **[개선 항목]**: (구체적인 개선 효과)

**4. ✅ 추천 개선 순서**
1. (가장 시급하고 효과적인 것부터 나열)
